## Architecture

- Theme: Vessel's actual code lives in this repo at root (`layout/`, `sections/`, `blocks/`, `snippets/`, `templates/`, `config/`, `locales/`, `assets/`) — originally imported from a manual theme export (.zip) after Theme Access token and an earlier GitHub-connect attempt both failed (see Known Constraints / Gotchas).
- **Live sync now working — and it's bidirectional, not one-way.** Vessel's own per-theme "Connect to GitHub" (distinct from the Admin-level one that previously created a duplicate empty theme) is connected to this repo's `main` branch. Earlier connection attempts failed with a "needs a valid theme in the branch" error simply because `main` didn't yet have the theme files merged in — once merged, the connection succeeded. **Correction of an earlier note in this file**: pushes to `main` sync to the live theme, *and* edits made in Shopify's theme editor get auto-committed back to `main` by `shopify[bot]` (commit message "Update from Shopify for theme ..."). This creates a real two-way sync loop: if someone edits the theme in Admin around the same time a repo push hasn't propagated yet, the bot's next auto-commit can capture and re-push whatever is live at that moment, potentially colliding with or overwriting an in-flight repo change. **Practical rule going forward: always `git pull origin main` immediately before editing theme JSON in the repo**, to catch any `shopify[bot]` commits first — merge conflicts in `templates/*.json` should be resolved in favor of the bot's version by default, since it reflects deliberate live/Admin-side edits, not stale repo state.
- Confirmed: Vessel's `product.json` template already includes `variant-picker`, `quantity`, and `buy-buttons` blocks in its default product section (`blocks/variant-picker.liquid` etc., referenced in `templates/product.json`'s `main` section, block `product-details`). This means the Roast Profile / Quantity selection UX needed for the subscription plan builder is handled natively — no custom Liquid was needed for that piece.
- `templates/index.json` (homepage) has been rebranded: hero copy, and the two "Featured collection" sections now point at real collection handles (`coffee-subscriptions`, `gifts`) instead of Vessel's demo "all" placeholder. Two empty "featured product" demo sections (no product assigned) were removed rather than left blank.
- Subscription logic (recurring billing, customer portal) lives in the subscription app (Skio, pending confirmation), integrated into the theme via app blocks/sections — not built custom in Liquid.
- Store data/catalog changes (products, collections, pricing, pages) go through the Shopify MCP connector, independent of theme choice.
- Navigation menus (main + footer) are wired up via the Shopify MCP connector's `menuUpdate` GraphQL mutation (store-level resource, not theme Liquid): main menu = Home, Our Coffee, Subscriptions (→ `coffee-subscriptions`), Decaf, Gifts, How It Works, FAQ; footer = About Us, Brew Guides, FAQ, Contact Us, Terms & Conditions, Privacy Policy, Cookie Policy, Accessibility Statement, Your Privacy Choices (CCPA), Search. Since the linked pages are still unpublished drafts (see Site Map), these links will 404 for visitors until the pages are published — same non-live status as the rest of the catalog/content.
- **Placeholder imagery added** (all products/collections/hero had zero images previously — `featuredMedia: null` across the board): free-license Unsplash photos attached via `fileCreate` (theme hero) and the Shopify image tools (products/collections). All 6 products, all 3 real collections (coffee-subscriptions, decaf, gifts), and the homepage hero (`image_1`/`image_2` in `templates/index.json`, referencing `photo-1511537190424-bbbab87ac5eb.jpg` and `photo-1565650839149-2c48a094196c.jpg`) now have images. These are stock placeholders, not real product photography — swap for real photos before public launch (flagging here, not silently treating as final).
- No logo image is set (header renders shop name as text) — a real logo requires actual brand design assets, not something to fabricate with stock photography.
- SEO title/description set on all 6 products and 3 real collections via `productUpdate`/`collectionUpdate`'s `seo` field (none had any set previously). Content pages don't expose a direct `seo` field on the `Page` type — that's handled via `global.title_tag`/`global.description_tag` metafields instead — deferred until pages are actually published (see Work Packages).
- **GitHub→Shopify sync direction can lag or appear inconsistent** — confirmed the repo→store direction synced once after an early merge, then didn't visibly pick up two subsequent pushes for an extended period; separately, a manual "pull latest" in Admin also didn't bring in a hero image change. The live theme (`OnlineStoreTheme` with `role: MAIN`, named `9.-High-Ground-Coffee-Club/main` — a separate theme entity from the original "Vessel" install, which is now `role: UNPUBLISHED`) can be inspected directly via `themes(first: N) { nodes { id name role updatedAt } }` and its files via `node(id) { ... on OnlineStoreTheme { files(filenames: [...]) { nodes { filename body { ... on OnlineStoreThemeFileBodyText { content } } } } } }` — useful for confirming what's actually live before assuming a push landed. **Writes to the live/MAIN theme via `themeFilesUpsert` are blocked by the Shopify MCP connector** (only unpublished/dev themes are writable that way), so there's no API-level way to force a resync from this side. To check sync health directly: look at `git log` on `main` for commits authored by `shopify[bot]` ("Update from Shopify for theme ...") — their presence/timestamps show when the store last pushed back to the repo (see bidirectional-sync note in Architecture above), which is the most reliable signal available, more so than the Admin UI's "pull latest" button.
- Found and fixed several leftover Vessel demo-content blocks that had nothing to do with coffee: a product-page "Embracing small joys" section with home-decor/furniture copy (now "Freshly roasted, just for you" with a coffee-at-home lifestyle image), and a product-page accordion titled "CARE & MAINTENANCE" with furniture care instructions (now "STORAGE & FRESHNESS" with actual coffee storage guidance). Worth another pass over other templates (page.json, cart.json, etc.) if more demo copy turns up later — this was found via a repo-wide text-content scan, not a guarantee of full coverage.

## Environment & Setup

- Shopify store connected via Shopify MCP connector: **Brown Bag Coffee Club** (`brown-bag-coffee-club.myshopify.com`), Basic plan, USD, US/EDT
- MCP connector gives Claude scoped store access (products, collections, orders, customers, inventory, analytics, discounts, pages, and general Admin GraphQL) — no raw admin credentials exchanged. It has disconnected/reconnected several times this session; treat gaps as transient, not a sign the integration was removed.
- Theme Access app: attempted twice (two separately generated tokens), both rejected by Shopify with 401 on every endpoint tested (including the themes endpoint itself), against both the correct store domain and the token's originating store. Root cause undiagnosed — likely something about how the app installed. Not pursued further once the manual export path worked; worth revisiting if live theme sync becomes worthwhile.
- Shopify CLI is installed globally in this environment (`npm install -g @shopify/cli`) for when a working auth method (Theme Access token, or `shopify auth login` interactively) becomes available.
- MCP tools (Shopify and GitHub both) are **deferred**: they don't appear as directly callable until `ToolSearch` loads their schema, and a fresh system-reminder listing the current deferred tool names appears each time the MCP server (re)connects. When the Shopify server reconnects mid-session its tool names get a new random ID prefix (e.g. `mcp__Shopify__*` → `mcp__<uuid>__*`), which orphans any tool name cached from earlier in the conversation. If a connector "isn't found" or a call throws `InputValidationError`, the fix is: re-check the latest deferred-tools system-reminder for the current names, then call `ToolSearch` (e.g. `select:mcp__github__get_me`) before invoking — this is not a sign the integration was actually removed.

## Known Constraints / Gotchas

- Claude Code can write/customize theme code but cannot perform Shopify admin actions directly (store creation, app installation, payment/tax setup) — those require the user to act in the Shopify dashboard, or to grant API/CLI access.
- Reference site's standard (non-promotional) pricing and full FAQ content are JS-rendered and weren't retrievable via WebFetch's HTML summarizer — worked around by pulling the raw catalog from Shopify's public `/products.json` endpoint instead (see Product Catalog below), which returns real prices directly.
- Atlas's live store contains many near-duplicate products (ad-campaign-specific variants, e.g. tagged `facebook-2023`, plus a straight "10% off" line) with different prices for what's otherwise the same product. The pricing table below uses the variant that appeared identically across multiple non-discounted listings, treated as their standard baseline — not a guarantee it's their exact current sticker price.
- Atlas does not sell specific named single-origin coffees as separate SKUs — the origin rotates monthly and is communicated via the postcard/tasting notes insert, not as a selectable product option. The purchasable variant axis is **Roast Profile** (Smooth & Balanced / Bright & Expressive), not origin. Our v1 catalog mirrors this structure; origin copy is placeholder/rotating content, not a product field.
